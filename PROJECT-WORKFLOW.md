# Atlasinn Codex Project Workflow

## Purpose

Keep Atlasinn work compact, organized, efficient, and validation-first.

## Chat Output

- Prefer one clean copyable Markdown block for project instructions, design notes, planning, database schemas, and workflow commands.
- Avoid repeated file dumps during small edits.
- Avoid long status messages, repeated summaries, repeated server links, or unnecessary large code outputs.
- Summarize changes clearly after the task is complete.

## File Work

- Work in the correct project folder and avoid changing unrelated files.
- Batch small edits instead of repeatedly rewriting the same file.
- Preserve existing structure unless the task requires a change.
- Do not resend or regenerate the same file repeatedly after minor edits unless requested or necessary for review.
- For Markdown/documentation files, include a title, table of contents when useful, headings, references, and source notes when applicable.
- For medical files, include source awareness and clinician-review framing.

## Local Preview Server

- Keep one local preview server running during an editing session when possible.
- Use one stable iPhone preview link during the task.
- Do not repeatedly start, stop, or check the server unless needed.
- Do not resend the same iPhone server link after every small edit.
- Only resend the link if the server or port changes, the user asks for it, or the task is complete.
- For small edits, say: "Updated. Refresh the same iPhone link."

## Validation Before Completion

- Review edited files.
- Validate structure and formatting.
- Check links, references, assets, and obvious syntax issues when applicable.
- Visualize UI/design changes when applicable, but batch screenshots and visual checks after meaningful groups of edits.
- Avoid unnecessary screenshots, image analysis, large file reads, or repeated browser validation unless needed to complete the task safely.
- Report what changed, what was validated, and what could not be checked.

## Usage-Control Rule

Minimize unnecessary Codex usage by batching edits, keeping the preview server stable, avoiding repeated link messages, avoiding repeated full-file outputs, and validating after meaningful progress rather than after every minor change.
